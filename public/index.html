New Service Auth Workflow

  1. Create or confirm the service in auth admin. Use a stable serviceKey such as todo, topic, or travel.
  2. Add permission definitions for that service. Each account receives one service permission, and the target service decides how to branch on it.
  3. If users log in to that service, create an OIDC client and register the service callback URL as a redirect URI.
  4. If the service backend calls auth internal APIs, create a Service Credential for that service with the minimum scopes needed, usually account.search.
  5. Copy the generated keyId and one-time secret. Store them only in the backend runtime environment, never in browser code.
  6. For root docker compose, map those values through service-prefixed env vars, for example TODO_AUTH_SERVICE_KEY_ID and TODO_AUTH_SERVICE_SECRET.
  7. Restart only the backend that consumes the credential. Rotate the credential if the one-time secret is lost or suspected to be exposed.

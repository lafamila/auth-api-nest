새 서비스 인증 연동 흐름

  1. 서비스 팀은 먼저 공개 service onboarding request API 로 serviceKey, 권한 정의, OIDC client, service credential spec 을 제출합니다.
  2. superadmin 은 Service Onboarding Requests 에서 요청 spec 을 검토하고 approve 또는 reject 합니다. approve 시에만 auth-side service spec 이 반영됩니다.
  3. approve 응답에서 OIDC client secret 또는 service credential secret 이 생성되면 이 화면에 한 번만 표시됩니다. secret manager 또는 backend env 로 즉시 옮기고 브라우저에는 남기지 않습니다.
  4. 각 active 서비스는 visitor permission definition 을 갖지만 계정 생성이나 서비스 승인 시 전체 조합 visitor row 를 벌크 생성하지 않습니다.
  5. 사용자가 특정 서비스에 처음 OIDC 로그인할 때만, 아직 row 가 전혀 없는 경우에 한해 visitor assignment 가 lazy 생성됩니다. revoked 또는 disabled row 는 자동 복구되지 않습니다.
  6. 권한 상승은 Account Access Requests 에서 approve/reject 하고, Account-Service Permission Dashboard 는 read-only 운영 현황판으로 유지됩니다.

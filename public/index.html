새 서비스 인증 연동 흐름

  1. 먼저 Services 에 새 서비스를 등록합니다. 이 값은 각 서비스가 auth 에 자신을 식별할 때 쓰는 안정적인 serviceKey 입니다.
  2. 해당 서비스에서 사용할 권한 레벨을 Permission Definitions 로 등록합니다. 계정 하나는 서비스 하나에 대해 한 개의 권한 레벨을 받고, 실제 분기는 각 서비스가 처리합니다.
  3. 브라우저 로그인/OIDC callback 이 필요한 서비스라면 OIDC Client 를 만들고 redirect URI 를 등록합니다.
  4. 서비스 backend 가 auth 의 internal API 를 호출해야 한다면 Service Credential 을 발급합니다. 필요한 scope 만 최소로 선택합니다.
  5. 발급된 keyId 와 1회 표시되는 secret 은 해당 서비스 backend 의 환경변수 또는 secret manager 에만 저장합니다. 프론트 코드에는 절대 넣지 않습니다.
  6. 환경변수 이름은 서비스별 prefix 를 붙여 구분합니다. 예: {SERVICE}_AUTH_SERVICE_KEY_ID, {SERVICE}_AUTH_SERVICE_SECRET
  7. credential 을 소비하는 backend 만 재시작하거나 환경변수를 다시 로드합니다. secret 을 잃어버렸거나 노출이 의심되면 기존 값을 조회하지 말고 rotate 합니다.
  8. 마지막으로 Permission Dashboard 에서 계정별 서비스 권한을 부여하거나 revoke 하고, 권한 없음 화면은 각 서비스 정책에 맞춰 처리합니다.
